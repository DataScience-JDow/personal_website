# Portfolio Maintenance

Use this checklist whenever a personal project is created or materially changed.

## Goal

The portfolio should reflect recruiter-relevant proof, not just a static list of tools or projects. If a personal build demonstrates meaningful product, data, platform, or AI engineering ability, evaluate whether it deserves a homepage mention, a deeper proof page, or both.

## Update Triggers

- A personal project gains a new workflow, automation, or integration that shows stronger system design.
- A project reaches a cleaner public-safe state with redacted screenshots, seeded demo data, or a shareable repo.
- Performance, reliability, deployment, or architecture work creates better proof than the current portfolio copy.
- A build demonstrates an ability the site currently undersells, such as backend design, ML implementation, data modeling, analytics engineering, or applied AI workflow design.

## Minimum Review Questions

1. Does this project strengthen Jeremy's case for data science, analytics engineering, data platform, BI engineering, or applied AI roles?
2. Is the current homepage case-study summary still accurate?
3. Should the project be added to or updated in `src/lib/portfolio.ts`?
4. Does the project deserve a deeper public proof page under `src/app/projects/[slug]/page.tsx`?
5. Is there any new public-safe artifact that should be exposed, such as screenshots, schema notes, architecture diagrams, benchmark results, or repo links?

## Preferred Outcome

When a project is relevant, update the portfolio in the same round of work rather than leaving the website stale.
